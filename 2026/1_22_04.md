# AWS PrivateLink Design Proposal for Overlapping CIDR Environments

## Objective

This document proposes a secure and scalable connectivity approach between two AWS VPCs where the CIDR ranges overlap.

Because the VPC CIDRs overlap, traditional networking options such as:

* VPC Peering
* AWS Transit Gateway route propagation
* Site-to-Site VPN routing between VPCs

are not suitable.

To address this limitation, the recommended approach is to use AWS PrivateLink.

AWS PrivateLink allows communication between services across VPCs and AWS accounts without requiring direct network connectivity, route exchange, or non-overlapping CIDR ranges.

---

## Proposed Architecture

### Consumer Side

* AWS Account A
* VPC1
* Application hosted in Amazon EKS Fargate

### Provider Side

* AWS Account B
* VPC2
* EC2 instance hosting the receiving application/service

### Communication Flow

1. The service running on the EC2 instance in VPC2 is exposed through an internal Network Load Balancer (NLB).
2. An AWS PrivateLink Endpoint Service is created using the internal NLB.
3. In VPC1, an Interface VPC Endpoint is created to consume the PrivateLink service.
4. The EKS application sends traffic to the Interface Endpoint DNS name.
5. Traffic remains entirely within the AWS private network.

---

## Why AWS PrivateLink is Recommended

AWS PrivateLink is the preferred approach in this case because:

* It supports overlapping CIDR ranges
* No VPC peering is required
* No route table exchange is required
* No NAT translation between VPCs is required
* Traffic remains on the AWS backbone network
* Communication can be restricted to only the required service and port
* Works across AWS accounts
* Provides a more secure and controlled access model

## Detailed Implementation Steps

### Step 1: Prepare the Application in VPC2

The receiving application hosted on the EC2 instance must:

* Listen on a fixed port such as 443, 8443, or 8080
* Listen on the EC2 private IP and not only on localhost
* Be reachable from within the VPC

Example validation:

```bash
curl http://<ec2-private-ip>:8080/health
```

---

### Step 2: Create a Target Group

In the provider account:

* Create a target group
* Target type: Instance
* Protocol: TCP or HTTP
* Port: Application port
* Register the EC2 instance as a target

Health checks should be configured appropriately:

* TCP health check for simple TCP services
* HTTP health check using a path such as `/health`

---

### Step 3: Create an Internal Network Load Balancer

In the provider account:

* Create a Network Load Balancer
* Scheme: Internal
* Deploy in private subnets
* Add a listener on the application port
* Forward traffic to the target group

Important:

* A security group should be associated during NLB creation
* AWS does not allow adding a security group later if the NLB is created without one

---

### Step 4: Configure Security Groups

#### NLB Security Group

Allow inbound traffic from the expected client range or from the consumer VPC.

Example:

* Inbound TCP 8080 from the consumer VPC CIDR or application security group

#### EC2 Security Group

Allow traffic from the NLB security group.

Example:

* Inbound TCP 8080 from NLB Security Group

---

### Step 5: Create a PrivateLink Endpoint Service

In the provider account:

* Create an Endpoint Service
* Select the internal NLB
* Enable manual acceptance if required

AWS will generate a service name similar to:

```text
com.amazonaws.vpce.us-east-1.vpce-svc-xxxxxxxxxxxxxxxxx
```

---

### Step 6: Grant Access to the Consumer Account

The provider account must explicitly allow the consumer account to connect.

Example principal:

```text
arn:aws:iam::<Consumer-Account-ID>:root
```

---

### Step 7: Create an Interface Endpoint in VPC1

In the consumer account:

* Create a VPC Endpoint
* Endpoint type: Interface Endpoint
* Select the PrivateLink service name shared by the provider
* Deploy the endpoint in private subnets
* Attach a dedicated security group

The Interface Endpoint creates private Elastic Network Interfaces (ENIs) inside the selected subnets.

---

### Step 8: Configure Endpoint Security Group

The Interface Endpoint security group should allow inbound traffic from the EKS workload.

Example:

* Inbound TCP 8080 from EKS pod CIDR or worker security group

---

### Step 9: Approve the Endpoint Connection

If manual acceptance is enabled, the provider account must approve the endpoint connection request.

Once approved, the endpoint status changes to Available.

---

### Step 10: Use the Endpoint DNS Name

The consumer account will receive a DNS name for the Interface Endpoint.

Example:

```text
vpce-xxxxxxxxxxxxxxxxx-abc.vpce-svc-xxxxxxxxxxxxxxxxx.us-east-1.vpce.amazonaws.com
```

The EKS application should use this DNS name to communicate with the provider service.

Example:

```yaml
env:
  - name: TARGET_URL
    value: "http://vpce-xxxxxxxxxxxxxxxxx-abc.vpce-svc-xxxxxxxxxxxxxxxxx.us-east-1.vpce.amazonaws.com:8080/upload"
```

## Key Advantages for the Client

This design provides the following business and technical advantages:

1. Supports overlapping CIDR ranges without redesigning either VPC
2. Avoids complex routing, NAT, or Transit Gateway dependencies
3. Reduces attack surface by exposing only a specific service
4. Provides private connectivity without traversing the public internet
5. Supports cross-account communication securely
6. Allows independent management of both environments
7. Scales easily if additional consumer VPCs are added in future

---

## Important Considerations

The following items should be validated during implementation:

* EC2 application is listening on the correct interface and port
* Target group health checks are successful
* NLB is internal and deployed in private subnets
* Security groups allow the required traffic
* Endpoint Service permissions include the consumer account
* Endpoint connection request is approved
* EKS pods can resolve and connect to the Interface Endpoint DNS
