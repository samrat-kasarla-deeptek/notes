Yes. For your case, the clean design is:

* **Account B / VPC2**: host the **EC2 service**
* **Account B / VPC2**: create an **internal NLB** and an **endpoint service**
* **Account A / VPC1**: create an **interface VPC endpoint**
* **Your app in EKS Fargate** sends data to the **endpoint DNS name** from VPC1

That works across accounts, keeps traffic on the AWS network, and does not need peering or route exchange between the VPCs. In PrivateLink, the **consumer creates the VPC endpoint**, the **provider creates the endpoint service**, and the **consumer initiates the connection**. ([AWS Documentation][1])

## What you need first

Before starting, have these ready:

* **Account A**: EKS cluster running in **private subnets** in VPC1
* **Account B**: EC2 instance in VPC2 running the app and listening on a fixed port such as `443`, `8443`, or `8080`
* At least **one subnet per AZ** you want to use on both sides
* The AWS account ID of each side
* A decision on the application port and health check path/port

PrivateLink endpoint services are backed by a **load balancer on the provider side**, and the consumer connects using an **interface endpoint**. ([AWS Documentation][1])

## End-to-end flow

### Step 1: Prepare the EC2 service in Account B / VPC2

On the EC2 instance in VPC2:

* Install or confirm your receiving application is running
* Make it listen on `0.0.0.0:<port>` and not only `localhost`
* Confirm it works from another host inside VPC2

Example:

```bash
curl http://<ec2-private-ip>:8080/health
```

This is important because the NLB will forward traffic to a target group, and the EC2 instance must be reachable on that target port. ([AWS Documentation][2])

---

### Step 2: Create a security group for the NLB in Account B

When creating the NLB, attach a security group from the start. AWS notes that if you create a Network Load Balancer **without** a security group, you **can’t add one later**. AWS also recommends associating a security group when you create it. ([AWS Documentation][3])

Example NLB SG:

* Inbound: allow TCP `8080` from **VPC1 CIDR** or your expected client range
* Outbound: allow all, or at least to the EC2 target port and health check port

---

### Step 3: Create the target group in Account B

In **EC2 → Target Groups**:

* Create target group
* Target type: **Instances** for a normal EC2 backend
* Protocol: `TCP` or `HTTP`
* Port: your app port, for example `8080`
* VPC: VPC2
* Register the EC2 instance

For health checks:

* Use `TCP` if your app is simple TCP
* Use `HTTP` and path like `/health` if your app supports it

NLB listeners forward traffic to target groups, so this is the object that points to your EC2 instance. ([AWS Documentation][2])

---

### Step 4: Create the internal NLB in Account B

In **EC2 → Load Balancers → Create Load Balancer → Network Load Balancer**:

Set:

* **Scheme**: `Internal`
* **VPC**: VPC2
* Choose **private subnets**
* Listener: `TCP` on your application port, for example `8080`
* Forward to the target group from Step 3
* Attach the security group from Step 2

An endpoint service requires a provider-side load balancer, and the service provider must specify a load balancer when creating the endpoint service. ([AWS Documentation][1])

---

### Step 5: Update the EC2 instance security group in Account B

On the EC2 instance SG, allow inbound traffic from the **NLB security group** on:

* the **target port**
* the **health check port** if different

AWS explicitly recommends referencing the NLB’s security group from the target’s security group so the targets accept traffic from the NLB but not directly from clients. ([AWS Documentation][3])

Example:

* Inbound TCP `8080` from `sg-nlb`
* Inbound TCP `8080` from `sg-nlb` for health checks too, if same port

---

### Step 6: Verify the NLB works inside Account B

From another EC2 in the same VPC, test:

```bash
nslookup <internal-nlb-dns>
curl http://<internal-nlb-dns>:8080/health
```

Also confirm in AWS:

* NLB status = Active
* Target group target = Healthy

If this fails, do not move ahead yet.

---

### Step 7: Create the endpoint service in Account B

In **VPC → Endpoint services**:

* Create endpoint service
* Select the **internal NLB**
* Choose whether **Acceptance required** should be on

After creation, AWS gives you a **service name**, something like:

```text
com.amazonaws.vpce.us-east-1.vpce-svc-xxxxxxxxxxxxxxxxx
```

The provider must then:

* add permissions for the consumer
* share the service name
* accept the connection request if manual acceptance is enabled. ([AWS Documentation][4])

---

### Step 8: Allow Account A to connect

In the endpoint service permissions, add the consumer account as an allowed principal.

Example:

```text
arn:aws:iam::<AccountA-ID>:root
```

By default, an endpoint service is **not available** to consumers. You must add permissions that allow specific AWS principals to create the interface endpoint. ([AWS Documentation][5])

---

### Step 9: Create a security group for the interface endpoint in Account A

In VPC1, create a security group for the interface endpoint ENIs.

Example:

* Inbound TCP `8080` from the EKS Fargate pod CIDR or the security group used by the workload path that will connect
* Outbound allow all, or at least to the provider port

Remember: an interface endpoint creates **endpoint network interfaces** in the consumer subnets you choose. ([AWS Documentation][1])

---

### Step 10: Create the interface endpoint in Account A / VPC1

In **VPC → Endpoints → Create endpoint**:

* Endpoint type: **Endpoint services that use NLBs**
* Paste the **service name** from Account B
* Select **VPC1**
* Select the **private subnets** where your EKS Fargate workload runs or can reach
* Attach the endpoint SG from Step 9

AWS creates an **endpoint network interface** in each selected subnet. That ENI is the private entry point from VPC1 to the service in VPC2. ([AWS Documentation][6])

---

### Step 11: Accept the endpoint connection in Account B

If you enabled manual acceptance:

* Go to **VPC → Endpoint services**
* Open the service
* Under **Endpoint connections**, approve the request

When the provider accepts, the VPC endpoint can move to **Available** and start being used. ([AWS Documentation][1])

---

### Step 12: Get the endpoint DNS name in Account A

Open the interface endpoint and copy the DNS name, which will look like:

```text
vpce-xxxxxxxxxxxxxxxxx-abc.vpce-svc-xxxxxxxxxxxxxxxxx.us-east-1.vpce.amazonaws.com
```

Your app in EKS Fargate should send data to this DNS name. Interface endpoints resolve traffic using DNS. ([AWS Documentation][1])

---

### Step 13: Update your EKS app config to use that DNS

In your Deployment or ConfigMap, point the destination to the PrivateLink endpoint DNS.

Example:

```yaml
env:
  - name: TARGET_URL
    value: "http://vpce-xxxxxxxxxxxxxxxxx-abc.vpce-svc-xxxxxxxxxxxxxxxxx.us-east-1.vpce.amazonaws.com:8080/upload"
```

Your pods then send traffic privately from VPC1 to the endpoint ENI in VPC1, across PrivateLink, into the NLB in VPC2, and finally to the EC2 instance. Traffic between the VPC endpoint and the endpoint service stays within the AWS network. ([AWS Documentation][1])

---

## What is specific to EKS Fargate here

On the **consumer side**, because your app is just making outbound calls, you usually do **not** need a Kubernetes `Service` or NLB in Account A. Your pod simply connects to the interface endpoint DNS name.

The EKS-specific NLB annotations matter when **EKS is the provider side**. For Fargate, AWS says NLBs for Fargate-backed pods must use **IP targets**, not instance targets. ([AWS Documentation][7])

So for **your** design:

* **Account A / EKS Fargate** = consumer only
* **Account B / EC2** = provider only

That means no AWS Load Balancer Controller is required in Account A just to consume PrivateLink.

## Network checks that commonly break this setup

Check these if it does not work:

1. **EC2 app only listening on localhost**

   * It must listen on the instance’s network interface, not only `127.0.0.1`.

2. **NLB target unhealthy**

   * Usually wrong health check port/path, or EC2 SG blocks NLB traffic.

3. **NLB security group missing**

   * Best set at creation time; AWS says you cannot attach one later if you created the NLB without one. ([AWS Documentation][3])

4. **Endpoint service permissions missing**

   * The consumer account must be explicitly allowed. ([AWS Documentation][5])

5. **Endpoint connection still pending**

   * If manual acceptance is on, the provider must approve it. ([AWS Documentation][1])

6. **Pod egress blocked**

   * Make sure the EKS/Fargate networking path allows outbound TCP to the interface endpoint on the app port.

## Very short build order

1. In **Account B**, run app on EC2
2. Create **target group**
3. Create **internal NLB**
4. Attach **NLB security group**
5. Allow EC2 SG from NLB SG
6. Create **endpoint service**
7. Allow **Account A** principal
8. In **Account A**, create **interface endpoint**
9. Attach endpoint SG
10. Approve connection in **Account B**
11. Use the **endpoint DNS name** from your EKS app
